# 代码影响分析器

分析需求变更或代码修改对系统的影响范围。

## 触发场景

- 编辑已开发的需求时
- 用户说"这个改动会影响哪些地方"
- 用户说"评估一下风险"
- 评审时需要评估影响范围
- 重构或修改核心代码时

## 分析维度

### 1. 直接影响分析

分析变更直接涉及的文件和代码：

```
📁 直接影响

需要修改的文件：
├── internal/sys/model/xxx.go        [修改] 新增字段
├── internal/sys/store/xxx.go        [修改] 新增查询方法
├── internal/sys/biz/xxx.go          [修改] 新增业务逻辑
└── internal/sys/controller/v1/xxx.go [修改] 新增接口参数

代码行数估算：
- 新增：约 150 行
- 修改：约 50 行
- 删除：约 10 行
```

### 2. 间接影响分析

分析调用链上下游的影响：

```
📊 间接影响

调用链分析：
Controller.Create()
  └── Biz.Create()
        ├── Store.Create()
        └── Cache.Invalidate()  ← 需要更新缓存键

被调用方：
├── OrderBiz.List() - 调用了 DeptChannelService
├── DashboardBiz.GetStats() - 调用了 DeptChannelService
└── ReportBiz.Export() - 调用了 DeptChannelService

共享数据影响：
└── sys_dept 表被多处引用，字段变更需同步
```

### 3. 接口兼容性分析

分析 API 变更的兼容性：

```
🔌 接口兼容性

接口变更：
POST /api/v1/sys/dept
├── 新增参数：channel_ids (可选)
├── 兼容性：✅ 向后兼容（新参数可选）
└── 前端影响：需要新增渠道选择组件

GET /api/v1/sys/dept/:id
├── 新增响应字段：channels
├── 兼容性：✅ 向后兼容（新增字段）
└── 前端影响：可选展示渠道信息

破坏性变更：❌ 无
```

### 4. 数据迁移分析

分析数据库变更的影响：

```
💾 数据迁移

表结构变更：
├── 新增表：sys_dept_channel
│   └── 影响：无历史数据迁移
├── 修改表：无
└── 删除表：无

数据迁移需求：
└── 无需迁移历史数据

回滚方案：
└── DROP TABLE sys_dept_channel;
```

### 5. 风险评估

综合评估变更风险：

```
⚠️ 风险评估

风险等级：🟡 中等

风险点：
1. [中] 权限控制逻辑变更，需充分测试
2. [低] 缓存失效策略需要验证
3. [低] Dashboard 数据过滤可能影响性能

缓解措施：
1. 编写完整的单元测试
2. 进行性能压测
3. 灰度发布，逐步放量
```

### 6. 测试建议

基于影响分析给出测试建议：

```
🧪 测试建议

必测项：
- [ ] 新功能正常流程
- [ ] 权限控制验证
- [ ] 缓存失效验证

回归测试：
- [ ] 订单列表功能
- [ ] Dashboard 数据展示
- [ ] 部门管理原有功能

性能测试：
- [ ] 大数据量下的查询性能
```

```
🔌 接口兼容性

[数据库变更分析]

[风险等级和风险点]

🧪 测试建议
[测试建议列表]

═══════════════════════════════════════════════
```

## 分析原则

1. **全面性**：覆盖所有可能的影响点
2. **准确性**：基于代码实际调用关系
3. **可操作性**：给出具体的应对措施
4. **风险导向**：重点关注高风险变更
